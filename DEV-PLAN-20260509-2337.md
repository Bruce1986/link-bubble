## 1. C6 — Theme.Holo → Material 遷移

### 背景
8 處 `@android:style/Theme.Holo*` 散落在 `AndroidManifest.xml` + [styles.xml](Application/LinkBubble/src/main/res/values/styles.xml)。Holo 是 2013 年（API 21）就被 Material 取代的 deprecated 系統主題；雖然 Android 14 仍能跑，但已脫離系統行為（深色模式、動態色彩、字體縮放都失效），且未來版本可能正式移除。

### 影響範圍
盤點目前所有用到 Holo 的地方：

| 檔案 | 位置 | 用途 |
|---|---|---|
| AndroidManifest.xml | L41 (現為 38) | `application` 預設 theme |
| AndroidManifest.xml.template | 同上 | 同上 |
| styles.xml | L49 | `TransparentTheme` parent |
| styles.xml | L59 | `ActiveTheme` parent |
| styles.xml | L73 | `ContentThemeBase` parent |
| styles.xml | L82 | `MainServiceThemeLight` parent |
| styles.xml | L86 | `MainServiceThemeDark` parent |
| styles.xml | L108 | `SectionHeaderTheme` parent |

衍生影響：
- 所有 Activity 的 theme attribute
- 自訂 widget 引用 `?android:attr/textColorPrimary` 等舊屬性
- AlertDialog、PopupMenu 的視覺
- Setting screens 用 `android.preference.*`（已 deprecated 的 API）

### Phase A — Parent Swap（最低風險）
**目標**：所有 6 個 style 的 `parent` 從 `android:Theme.Holo*` → `Theme.MaterialComponents.*`，視覺差異最小化。

**改動**：
```xml
<!-- styles.xml 範例 -->
<style name="TransparentTheme" parent="Theme.MaterialComponents.Light.NoActionBar">
    <item name="android:windowBackground">@android:color/transparent</item>
    <item name="android:windowIsTranslucent">true</item>
    ...
</style>

<style name="MainServiceThemeLight" parent="Theme.MaterialComponents.Light">
    ...
</style>

<style name="MainServiceThemeDark" parent="Theme.MaterialComponents">
    ...
</style>
```

**Manifest 改動**：
```xml
<application
    android:theme="@style/AppTheme"  <!-- 從 @android:style/Theme.Holo.Light 改成 -->
    ...>
```
（因為 Holo 在 manifest 是 application 預設主題，現在每個 activity 都已經有自己的 `@style/AppTheme`，可以拿掉 application-level 的 Holo，或改成繼承 Material 的 AppTheme。）

**驗證點**（手動測試）：
- [ ] HomeActivity 開得起來、按鈕可按
- [ ] HistoryActivity 列表正常
- [ ] SettingsActivity 進得去
- [ ] EntryActivity（從別的 App 分享連結）能彈出
- [ ] ExpandedActivity（點開一個 bubble）正常
- [ ] 主題色不會變太離譜（可接受些微 padding/字體變動）

**rollback 條件**：如果視覺差異無法接受 → `git checkout` styles.xml 即可，影響不擴及程式碼。

**預估時間**：1 小時改 + 30 分鐘測試。

## 2. H2 — AsyncTask 替換為 ExecutorService

### 範圍
7 個檔案使用 `AsyncTask`：

| 檔案 | AsyncTask 子類 | 複雜度 |
|---|---|---|
| MainApplication.java | `InitBlackListCollectorAsyncTask`、`DownloadAdInsertionDataAsyncTask`（grep 確認） | 簡單 |
| YouTubeEmbedHelper.java | (待確認) | 簡單 |
| DownloadImage.java | (待確認) | 中等（有 progress） |
| ArticleContent.java | `BuildContentTask` | 中等 |
| LoadFaviconTask.java | (mozilla 移植) | 複雜（generic + cancellation） |
| BitmapUtils.java | (mozilla 移植) | 中等 |
| UiAsyncTask.java | (mozilla 移植，是個 wrapper) | 框架類，可能整個刪 |

### 設計：Util.runInBackground

在 [Util.java](Application/LinkBubble/src/main/java/com/linkbubble/util/Util.java) 加：

```java
public final class BackgroundExecutor {
    private static final int CORE_POOL_SIZE = 2;
    private static final int MAX_POOL_SIZE = 4;
    private static final long KEEP_ALIVE_SECONDS = 30;

    private static final ExecutorService sExecutor = new ThreadPoolExecutor(
            CORE_POOL_SIZE, MAX_POOL_SIZE,
            KEEP_ALIVE_SECONDS, TimeUnit.SECONDS,
            new LinkedBlockingQueue<>(),
            r -> {
                Thread t = new Thread(r, "LinkBubble-bg-" + sCounter.getAndIncrement());
                t.setPriority(Thread.NORM_PRIORITY - 1);
                return t;
            });

    private static final Handler sMainHandler = new Handler(Looper.getMainLooper());
    private static final AtomicInteger sCounter = new AtomicInteger();

    public interface BackgroundTask<T> {
        T run() throws Exception;
    }

    public interface UiCallback<T> {
        void onResult(T result);
    }

    public static <T> Future<?> runInBackground(BackgroundTask<T> bg, UiCallback<T> ui) {
        return sExecutor.submit(() -> {
            try {
                T result = bg.run();
                if (ui != null) sMainHandler.post(() -> ui.onResult(result));
            } catch (Exception e) {
                Log.e("BackgroundExecutor", "task failed", e);
                CrashTracking.logHandledException(e);
            }
        });
    }
}
```

### 替換策略
- **簡單 case**（fire-and-forget）：`new Foo().execute()` → `BackgroundExecutor.runInBackground(() -> initBlackListCollector(), null)`
- **有 callback case**：用 `UiCallback<T>` 接 onPostExecute
- **取消支援**：保留 `Future<?>`，呼叫端用 `future.cancel(true)` 取代 `task.cancel(true)`
- **mozilla 移植檔案**（`UiAsyncTask`、`LoadFaviconTask`）：可能不動，因為 favicon 模組是 plug-in 來的。先看完前 6 個再決定。

### 風險
- `LoadFaviconTask` 邏輯較複雜（cancellation + generic + chained loading），改錯會讓 favicon 顯示異常
- 建議分兩個 commit：(1) MainApplication / YouTubeEmbedHelper / DownloadImage / ArticleContent；(2) mozilla 系列

### 預估時間
2-3 小時 + 測試。

## 3. H3 — 依賴升級 Batch 1

### 目標
低風險、高價值的依賴升級，主要解決 jsoup CVE 與本地 jar Maven 化。

### 異動表

| 套件 | 從 | 到 | 改動類型 |
|---|---|---|---|
| jsoup | `libs/jsoup-1.7.3.jar` | `org.jsoup:jsoup:1.17.2`（Maven） | 安全修補 |
| picasso | `libs/picasso-2.1.1.jar` | `com.squareup.picasso:picasso:2.71828`（Maven） | Maven 化 |
| slf4j-android | `libs/slf4j-android-1.5.8.jar` | `org.slf4j:slf4j-android:1.7.36` | Maven 化 |
| legacy-support-v4 | `androidx.legacy:legacy-support-v4:1.0.0` | (移除 if unused) | 清理 |

### 步驟
1. **build.gradle 改 dependencies**
   ```gradle
   dependencies {
       // ... 保留其他
       implementation 'androidx.appcompat:appcompat:1.6.1'
       // ...
       implementation 'org.jsoup:jsoup:1.17.2'         // was libs/jsoup-1.7.3.jar
       implementation 'com.squareup.picasso:picasso:2.71828'  // was libs/picasso-2.1.1.jar
       implementation 'org.slf4j:slf4j-android:1.7.36' // was libs/slf4j-android-1.5.8.jar
       // 移除：implementation files('libs/jsoup-1.7.3.jar')
       // 移除：implementation files('libs/picasso-2.1.1.jar')
       // 移除：implementation files('libs/slf4j-android-1.5.8.jar')
   }
   ```

2. **刪除 libs 內舊 jar**
   - `git rm Application/LinkBubble/libs/jsoup-1.7.3.jar`
   - `git rm Application/LinkBubble/libs/picasso-2.1.1.jar`
   - `git rm Application/LinkBubble/libs/slf4j-android-1.5.8.jar`
   - 保留 `libs/Parse-1.4.1.jar`（沒換）

3. **修 jsoup API 變動**
   - jsoup 1.7 → 1.17 大跨度，可能有 API 改變。重點：`StringUtil` 移到 `org.jsoup.internal.StringUtil`（[ExpandedActivity.java:32](Application/LinkBubble/src/main/java/com/linkbubble/ui/ExpandedActivity.java:32) 引用了它）
   - 跑 build 看 compile 錯誤、逐個修

4. **檢查 legacy-support-v4 用途**
   ```bash
   grep -rn "android.support.v4\|androidx.legacy" Application/LinkBubble/src/main/java
   ```
   如果只剩下 androidx 引用沒到 legacy 的，可以把 `legacy-support-v4` dependency 刪掉

5. **build + 煙霧測試**

### 風險
- jsoup 9 個 minor 版號跨度，內部 API 多有變動。最常見的是 `StringUtil`、`Parser` 的 method signature
- picasso 2.1 → 2.71 中間 picasso 改寫過 cache 與 Loader 介面，但常用的 `Picasso.with(ctx).load(url).into(view)` 維持相容
- slf4j 升級若有舊版獨特行為被依賴會出問題，但這是 logger，影響面小

### 預估時間
2 小時，主要花在修 jsoup compile errors。

## 4. H6 — Firebase Crashlytics 接入指引

### 你需要做的事（在 Firebase Console / Google Cloud）

#### Step 1：建立 Firebase 專案
1. 前往 https://console.firebase.google.com/
2. 點 **Add project**，名稱輸入 `LinkBubble`（或你想要的）
3. 是否啟用 Google Analytics：**先不要**，如果之後要用 GA4 再啟。Crashlytics 不依賴 GA。
4. 完成建立。

#### Step 2：在專案下加入 Android App
1. 在專案首頁點 Android icon
2. **Android package name**：填 `com.linkbubble.playstore.dev`（debug build only）
   - 決議：只追 dev，不 release。原 `com.linkbubble.playstore` 不是你的 namespace，無法上架；未來若要自己上架需先改 `applicationId` 為你自己的（例如 `com.brucejhang.linkbubble`），那是另一個獨立工程
3. **App nickname**：自訂（例如「LinkBubble Dev」）
4. **Debug signing certificate SHA-1**（可選）：debug 用不需要；如果之後要做 Dynamic Links 才需要
5. 下載 `google-services.json`

#### Step 3：把 google-services.json 放到專案
- 路徑：`Application/LinkBubble/google-services.json`
- ⚠️ **不要 commit 到 git**！把它加進 `.gitignore`：
  ```gitignore
  Application/LinkBubble/google-services.json
  ```
- 如果你有兩個 product flavors（playstore/dev），Firebase 在 `google-services.json` 裡會把兩個 package name 都包含，一份檔案就夠

#### Step 4：開啟 Crashlytics
1. 在 Firebase Console 左邊選單找 **Crashlytics**
2. 點 **Get started** → 跟著精靈走
3. 完成後會看到「Waiting for first crash...」狀態

#### Step 5：把 google-services.json 路徑告訴我
你放好之後跟我說「放好了」，我會接手做 Step 6 之後的 Gradle / 程式整合。

### 我會幫你做的事（你說「放好了」之後）

#### Step 6：Gradle 整合
- 在 `Application/build.gradle`（root）加 buildscript classpath：
  ```gradle
  buildscript {
      dependencies {
          classpath 'com.google.gms:google-services:4.4.2'
          classpath 'com.google.firebase:firebase-crashlytics-gradle:3.0.2'
  ```

- 在 `Application/LinkBubble/build.gradle` 最下方加：
  ```gradle

  ```


   - 修法：在 `onCreate` 補上 `MainApplication.registerForBus(this, this)`，順帶啟用了 3 個 @Subscribe 事件

3. **Picasso `Transformation.transform() returned null`**
   - 原因：H3 Batch 1 升 picasso 後，新版禁止 transform 回傳 null（舊版容忍）
   - 涉及檔案：[BubbleView.java:181](Application/LinkBubble/src/main/java/com/linkbubble/ui/BubbleView.java:181)（FaviconTransformation）、[PageInspector.java:347](Application/LinkBubble/src/main/java/com/linkbubble/util/PageInspector.java:347)（TouchIconTransformation）
   - 修法：移除 `return null` 改回傳 bitmap；順帶把兩個 transform 的 `key()` 從同名 `"faviconTransformation()"` 拆開（之前 cache key 衝突）

4. **`FaviconCache` IllegalArgumentException: cannot use a recycled source**
   - 原因：我在 #3 修過程中加了 `source.recycle()` 太貪心；favicon bitmap 是跨層共享的，FaviconCache 還想用它
   - 修法：移除兩個 transform 內的 `source.recycle()`

5. **點 `target="_blank"` / `window.open` 連結 crash**：`New WebView for popup window must not have been previously navigated`
   - 原因：[WebViewRenderer.java:937 `onCreateWindow`](Application/LinkBubble/src/main/java/com/linkbubble/webrender/WebViewRenderer.java:937) 把已經 navigate 過的 WebView 交回 Chromium，新版 WebView 拒絕
   - 修法：改用「throwaway WebView 截 URL」pattern — 給 Chromium 一個全新 WebView，截到要 navigate 的 URL 後 destroy 它，URL 走 MainController 正常開新 tab 流程
   - 這原本被使用者列為「老問題」，但實際上是 C6/H3 之前就存在的 Chromium API 變動造成
