# Link Bubble 升級專案 Code Review

**日期**：2026-05-09 22:20
**Reviewer**：Claude (Opus 4.7)
**Scope**：整個專案，重點放在從 Support Library → AndroidX、SDK 28 → 34 升級遺留的問題

---

## 摘要

這是一個老舊（2014 年代）的 Android App，原本基於 Support Library 與 SDK 21 左右。專案經過 SDK 34 / AndroidX 升級後**目前可以編譯出 debug APK**，但仍留下相當多的技術債、安全弱點與運行期未爆彈。共找到 **34** 個具體問題：

| 嚴重度 | 數量 | 摘要 |
|---|---|---|
| 🔴 Critical | 6 | Native lib 缺失會 NPE、Release build 被註解、過時 deprecated 依賴、缺少 `android:exported`、明文流量、使用已淘汰主題 |
| 🟠 High | 13 | 大量 `new Handler()` 無 Looper、AsyncTask、API 棄用、文件不一致、AdBlock 失效 |
| 🟡 Medium | 12 | WebView 安全設定、Backup 風險、TODO/dead code、lint 設定可疑 |
| 🟢 Low | 3 | 文件冗餘、寫法習慣、輕微訊息不準 |

下方章節依嚴重度排序，每條都附上 `檔案路徑:行號`。

## 🔴 Critical（必須立即處理）

### C1. Native lib 缺失但 Java 端仍 `System.loadLibrary("LinkBubble")` → 啟動 AdBlock/TP 必 crash
- **位置**：[ABPFilterParser.java:18](Application/LinkBubble/src/main/java/com/linkbubble/adblock/ABPFilterParser.java:18)、[TPFilterParser.java:13](Application/LinkBubble/src/main/java/com/linkbubble/adblock/TPFilterParser.java:13)
- **問題**：兩個檔案都在 static initializer 呼叫 `System.loadLibrary("LinkBubble")`，但 [build.gradle:104-108](Application/LinkBubble/build.gradle:104) 把 `externalNativeBuild` 整段註解掉、`jni/` 目錄下大部分 `.cpp` 仍缺失（CMakeLists.txt 列了 9 個 cpp，僅 `main.cpp` 存在）、且 `jniLibs/` 沒有預先打包的 `.so`。
- **影響**：當 `MainApplication.onCreate()` 觸發 [`createABPParser()`、`createTrackingProtectionList()`](Application/LinkBubble/src/main/java/com/linkbubble/MainApplication.java:101) 時會丟 `UnsatisfiedLinkError`，AdBlock/TP 功能直接失效；`build_fix_summary.md` 提到曾「停用原生編譯」並未在最新 commit `ee0c3e9b "Restore native lib"` 後恢復一致。
- **建議**：擇一處理：(a) 取回完整 cpp 來源並重新啟用 `externalNativeBuild`；(b) 若不再支援，移除 `ABPFilterParser`/`TPFilterParser` 與相關設定 UI，並把 [MainApplication.java:101-108](Application/LinkBubble/src/main/java/com/linkbubble/MainApplication.java:101) 的 SDK_INT 判斷一併刪除。

### C2. Release buildType 被整段註解，無法產出 release APK
- **位置**：[build.gradle:97-101](Application/LinkBubble/build.gradle:97)
- **問題**：`buildTypes { release { ... } }` 整塊以 `//` 註解，等同退化為「只剩 debug」。`signingConfigs.release` 仍存在但永遠不會被引用；`build_fix_summary.md` 也記錄了「只能 build debug」。
- **建議**：取消註解，讓 ProGuard / signing / shrinkResources 重新接上；或在註解處明確標示「TODO: 重啟 release build」並建追蹤 issue。

### C3. README 與 build.gradle 的 SDK 版本完全不一致
- **位置**：[README.md:39](README.md:39) vs [build.gradle:46,53,54](Application/LinkBubble/build.gradle:46)
- **問題**：README 說「Update `compileSdkVersion`/`targetSdkVersion` 至 33、`minSdkVersion` 至 28」，實際 build.gradle 是 `compileSdkVersion 34`、`targetSdkVersion 34`、`minSdkVersion 21`。新成員照 README 操作會把版本改錯。
- **建議**：把 README 改寫為實際值（34/34/21），並說明為何 minSdk 留在 21（如果是相容性需求）。

### C4. 大量 Activity 缺 `android:exported`，Android 12+ 安裝會被拒
- **位置**：[AndroidManifest.xml:44, 64, 149, 154, 159, 164, 169, 207](Application/LinkBubble/src/main/AndroidManifest.xml:44)
- **問題**：`HomeActivity`(L44)、`HistoryActivity`(L64)、`SettingsActivity`(L149)、`SettingsDefaultAppsActivity`(L154)、`SettingsMoreActivity`(L159)、`SettingsHelpActivity`(L164)、`SettingsDomainsActivity`(L169)、`ExpandedActivity`(L207) **都沒有** `android:exported`。Android 12 (API 31) 起，凡是有 intent-filter 的元件必須顯式宣告，未宣告會在安裝時拋 `INSTALL_PARSE_FAILED_MANIFEST_MALFORMED`。雖然這幾個沒有 intent-filter，但 lint/build 仍會警告，且某些 OEM 行為不一致。
- **建議**：對全部沒設定的 activity 補 `android:exported="false"`；只有 `EntryActivity`(L77) 因為對外 receive 才應為 `true`（已正確設定）。

### C5. `usesCleartextTraffic="true"` 在瀏覽器 App 中是嚴重資安弱點
- **位置**：[AndroidManifest.xml:40](Application/LinkBubble/src/main/AndroidManifest.xml:40)
- **問題**：整個 application 開放明文 HTTP，任何 MITM 都能竄改頁面內容；瀏覽器類 App 尤其會被視為高風險。Google Play 對這類設定也會警告。
- **建議**：改為 `false`，並建立 `res/xml/network_security_config.xml`：對於需要 HTTP 的特定使用者輸入網址，由瀏覽器層另外處理 user warning，不應在 manifest 全域開放。

### C6. App 主題仍使用 2013 年棄用的 `Theme.Holo.Light`
- **位置**：[AndroidManifest.xml:41](Application/LinkBubble/src/main/AndroidManifest.xml:41)、[styles.xml:49,59,73,82,86,108](Application/LinkBubble/src/main/res/values/styles.xml:49)
- **問題**：`@android:style/Theme.Holo.Light` 自 API 21 (Lollipop) 起即被 `Material` 取代，目前 Android 14 仍能跑但行為已與系統脫節（深色模式、動態色彩、字體縮放、圓角 widget 完全失效），多 OEM 已預告會在未來版本停止支援系統主題定義。
- **建議**：遷移至 `Theme.Material3.Light`（已加入 `material:1.11.0`）或至少 `Theme.AppCompat.Light`，分批調整自訂 style 的 parent。

## 🟠 High

### H1. 13 處 `new Handler()` 沒帶 Looper，Android 12+ 會 deprecation warning，舊機背景執行會誤掛 worker thread
- **位置**：[GeckoBackgroundThread.java:29](Application/LinkBubble/src/main/java/org/mozilla/gecko/util/GeckoBackgroundThread.java:29)、[ExpandedActivity.java:71](Application/LinkBubble/src/main/java/com/linkbubble/ui/ExpandedActivity.java:71)、[BubbleView.java:290](Application/LinkBubble/src/main/java/com/linkbubble/ui/BubbleView.java:290)、[Prompt.java:46](Application/LinkBubble/src/main/java/com/linkbubble/ui/Prompt.java:46)、[EntryActivity.java:35](Application/LinkBubble/src/main/java/com/linkbubble/ui/EntryActivity.java:35)、[ContentView.java:1201,2000](Application/LinkBubble/src/main/java/com/linkbubble/ui/ContentView.java:1201)、[SettingsDomainsActivity.java:132](Application/LinkBubble/src/main/java/com/linkbubble/ui/SettingsDomainsActivity.java:132)、[SettingsActivity.java:135](Application/LinkBubble/src/main/java/com/linkbubble/ui/SettingsActivity.java:135)、[PageInspector.java:71](Application/LinkBubble/src/main/java/com/linkbubble/util/PageInspector.java:71)、[AppPoller.java:110](Application/LinkBubble/src/main/java/com/linkbubble/util/AppPoller.java:110)、[WebViewRenderer.java:99,414](Application/LinkBubble/src/main/java/com/linkbubble/webrender/WebViewRenderer.java:99)
- **問題**：`new Handler()` 自 API 30 起 deprecated，行為依呼叫 thread 的 Looper 而定，可能誤綁到非預期 thread。
- **建議**：UI 用 `new Handler(Looper.getMainLooper())`，背景用顯式建立的 `HandlerThread.getLooper()`。`lint.xml:6-9` 目前壓抑了 `HandlerLeak`，更顯得這條風險被刻意忽略。

### H2. 7 個檔案仍使用 deprecated `AsyncTask`
- **位置**：[MainApplication.java:22,159](Application/LinkBubble/src/main/java/com/linkbubble/MainApplication.java:22)、[YouTubeEmbedHelper.java](Application/LinkBubble/src/main/java/com/linkbubble/util/YouTubeEmbedHelper.java)、[DownloadImage.java](Application/LinkBubble/src/main/java/com/linkbubble/util/DownloadImage.java)、[ArticleContent.java](Application/LinkBubble/src/main/java/com/linkbubble/articlerender/ArticleContent.java)、[LoadFaviconTask.java](Application/LinkBubble/src/main/java/org/mozilla/gecko/favicons/LoadFaviconTask.java)、[BitmapUtils.java](Application/LinkBubble/src/main/java/org/mozilla/gecko/gfx/BitmapUtils.java)、[UiAsyncTask.java](Application/LinkBubble/src/main/java/org/mozilla/gecko/util/UiAsyncTask.java)
- **問題**：`AsyncTask` 在 API 30 棄用且行為混亂（serial executor、洩漏 Activity 等），這個 App 以瀏覽器與 service 為主，AsyncTask 在 background service 中執行尤易掛掉。
- **建議**：替換為 `ExecutorService` + `Handler.post`，或最低工程量的 `ListenableFuture`/`Coroutines`（如願引入 Kotlin）。先處理在主流程 hot path 上的 `InitBlackListCollectorAsyncTask`、`DownloadAdInsertionDataAsyncTask`、`LoadFaviconTask`。

### H3. 過時/已停止維護的依賴版本
- **位置**：[build.gradle:131-144](Application/LinkBubble/build.gradle:131)
- **問題**：`retrofit:1.5.0`（2013，已被 retrofit2 取代且不支援 OkHttp4+）、`butterknife:7.0.0`（2015，停止維護，AGP8 下 R class 行為微變動）、`otto:1.3.8`（Square 自己 deprecate，建議用 LiveData/Flow/EventBus 3）、`stickylistheaders:2.1.2`（不再維護）、本地 jar：`picasso-2.1.1.jar`、`jsoup-1.7.3.jar`、`slf4j-android-1.5.8.jar`（皆 2013 級老舊；jsoup 1.7.3 有已知 XSS 漏洞，CVE-2021-37714 等多項早期版本受影響）、`com.timehop.stickyheadersrecyclerview:0.4.0`（停更）。
- **建議**：建立明確的依賴升級計畫，最低限度 jsoup 升級至 ≥1.15.3，picasso 用 Maven 拉 2.71828，slf4j 升至 1.7.x or 2.x；retrofit/otto/butterknife 屬大範圍重構，可分階段。

### H4. 無效的 permission 名稱
- **位置**：[AndroidManifest.xml:5](Application/LinkBubble/src/main/AndroidManifest.xml:5)
- **問題**：`android.permission.ACTION_MANAGE_OVERLAY_PERMISSION` 不是 permission，而是 Intent action 名稱。系統會直接忽略這行宣告，但會在 lint/play console 顯示警告（unknown permission）。
- **建議**：刪掉這行。`Settings.ACTION_MANAGE_OVERLAY_PERMISSION` 應在 Java 用 Intent 觸發 `Settings.canDrawOverlays()` 流程。

### H5. `GET_TASKS` permission 自 API 21 棄用
- **位置**：[AndroidManifest.xml:7](Application/LinkBubble/src/main/AndroidManifest.xml:7)
- **問題**：`android.permission.GET_TASKS` 已被 `REAL_GET_TASKS`（system app only）和 `UsageStatsManager` 取代。在 minSdk 21 的設定下這個權限完全沒用。
- **建議**：刪除；如果還在用 `ActivityManager.getRunningTasks()` 應改用 `UsageStatsManager.queryUsageStats()`（需 PACKAGE_USAGE_STATS）或重新設計流程。

### H6. `MainApplication.onCreate` 仍 import 並呼叫被「停用」的 Analytics / CrashTracking
- **位置**：[MainApplication.java:42,43,96,130](Application/LinkBubble/src/main/java/com/linkbubble/MainApplication.java:42)；專案內共 137 處 `CrashTracking.*` / `Analytics.*` 呼叫
- **問題**：`Analytics.init()` 是空殼（[Analytics.java:13-14](Application/LinkBubble/src/main/java/com/linkbubble/util/Analytics.java:13)），其他 track 函式內容都被 `/* ... */` 註解掉；`CrashTracking` 只剩 Log 包裝。`build_fix_summary.md` 寫「註解掉所有 CrashTracking/Analytics」但實際是改成空殼，137 個呼叫點仍存在於商業邏輯關鍵路徑上。
- **建議**：要嘛真的接回 Crashlytics/GA/Firebase，要嘛把所有呼叫點清乾淨；保留空殼的最大代價是讀程式碼的人會以為功能還活著。

### H7. `MainService.onCreate` 不在 `onStartCommand` 第一個 5 秒內呼叫 `startForeground`
- **位置**：[MainService.java:108-123](Application/LinkBubble/src/main/java/com/linkbubble/MainService.java:108)
- **問題**：在 Android 8+，foreground service 必須在 `startForegroundService()` 後 5 秒內呼叫 `startForeground()`，否則 ANR/`ForegroundServiceDidNotStartInTimeException`。目前 `onCreate` 走 `setTheme` → `super.onCreate` → 大段邏輯後才到 `showDefaultNotification`。如果 `showDefaultNotification` 中有任何例外（例如 `NotificationCompat.Builder(context, channel)` 在 channel 失敗時），就會直接 crash。建議優先 startForeground、後續再做設定。
- **建議**：把 `showDefaultNotification()` 拉到 `onCreate` 最前面、緊接 `super.onCreate()` 之後，並把 `Settings.get()`、Bus 註冊延後。

### H8. `MainService.cancelCurrentNotification()` 用棄用的 `stopForeground(boolean)`
- **位置**：[MainService.java:230-233](Application/LinkBubble/src/main/java/com/linkbubble/MainService.java:230)
- **問題**：`stopForeground(true)` 自 API 33 棄用，應使用 `stopForeground(STOP_FOREGROUND_REMOVE)`。在 targetSdk 34 下會 lint warning，且未來行為可能改變。
- **建議**：替換為 `stopForeground(Service.STOP_FOREGROUND_REMOVE)`。

### H9. `NotificationCompat.Builder(this)` 使用無 channel 建構式
- **位置**：[MainApplication.java:145](Application/LinkBubble/src/main/java/com/linkbubble/MainApplication.java:145)、[MainService.java:305](Application/LinkBubble/src/main/java/com/linkbubble/MainService.java:305)
- **問題**：`NotificationCompat.Builder(Context)` 是 deprecated；在 Android 8+ 上若沒有 channel 通知不會顯示，目前 `showNewBraveBrowserHiddenNotification` 完全不會在 O+ 顯示通知。
- **建議**：使用 `NotificationCompat.Builder(context, channelId)`，並在 onCreate 之前建立 channel。

### H10. NotificationChannel 名稱出現 `"snap map fake location"`
- **位置**：[MainService.java:198,201](Application/LinkBubble/src/main/java/com/linkbubble/MainService.java:198)
- **問題**：通道名稱與 ID（`"snap map fake location "`、`"snap map channel"`）顯然是從 Stack Overflow / 別的專案複製貼上未改名。對使用者來說在系統設定中會看到一個與本 App 完全無關的中英混雜字串。
- **建議**：改為 `"LinkBubble"` / `"link_bubble_default"` 之類有語意的命名，並在 strings.xml 多語化。

### H11. `WebViewRenderer.setJavaScriptEnabled(true)` 沒設安全限制
- **位置**：[WebViewRenderer.java:124](Application/LinkBubble/src/main/java/com/linkbubble/webrender/WebViewRenderer.java:124)
- **問題**：對任意網頁開啟 JS 是必要的，但同時應顯式關閉 `setAllowFileAccess(false)`、`setAllowFileAccessFromFileURLs(false)`、`setAllowUniversalAccessFromFileURLs(false)`、`setSafeBrowsingEnabled(true)`，否則惡意網頁能透過 `file://` 跳出 sandbox。
- **建議**：在 124 行下方追加上述四個設定。`setSavePassword(false)` 已正確設定，繼續保持。

### H12. C++ 原始碼遺失與 build.gradle 的 `externalNativeBuild` 註解狀態不一致
- **位置**：[CMakeLists.txt:24-32](Application/LinkBubble/src/main/jni/CMakeLists.txt:24)（列出 9 個 cpp，僅 `main.cpp` 存在）vs [build.gradle:104-108](Application/LinkBubble/build.gradle:104)（externalNativeBuild 整段註解）
- **問題**：CMake 列表寫 `ABPFilterParser.cpp filter.cpp BloomFilter.cpp hashFn.cpp HashSet.cpp cosmeticFilter.cpp TPParser.cpp FirstPartyHost.cpp TrackerData.cpp`，但這些檔案都不在 jni/ 目錄。雖然 build.gradle 註解保住了 build，但留著矛盾的 CMakeLists 會誤導未來工程師。
- **建議**：`jni/` 完全移除，或補回 cpp 後正式啟用；至少在 CMakeLists.txt 上方加一行註解說明此檔已停用。

### H13. `clean.dependsOn showVersion` 異常 task 依賴
- **位置**：[build.gradle:147-153](Application/LinkBubble/build.gradle:147)
- **問題**：每次 `./gradlew clean` 都會印 buildscript classpath 版本到 console。看似輔助但讓 clean 變慢，且 `firstLevelModuleDependencies.moduleVersion` 在某些 AGP 版本（含 8.x）已不保證可解析。
- **建議**：刪除或改為 `task showVersion { ... } /* 自行 ./gradlew showVersion 呼叫 */`。

## 🟡 Medium

### M1. `allowBackup="true"` 配合無 `dataExtractionRules` 與 `fullBackupContent`
- **位置**：[AndroidManifest.xml:37](Application/LinkBubble/src/main/AndroidManifest.xml:37)
- **建議**：對瀏覽歷史 / cookie 的 App 改為 `false`，或定義 `android:dataExtractionRules`（API 31+）+ `android:fullBackupContent` 排除敏感資料。

### M2. `FOREGROUND_SERVICE_SPECIAL_USE` 需在 `<service>` 補 `<property>` 說明用途（Play Console 上架要求）
- **位置**：[AndroidManifest.xml:14-15,197-204](Application/LinkBubble/src/main/AndroidManifest.xml:14)
- **問題**：自 Android 14 起，`SPECIAL_USE` 必須附帶 `<property android:name="android.app.PROPERTY_SPECIAL_USE_FGS_SUBTYPE" android:value="..." />` 並在 Play 後台填寫用途。目前缺失，未來上架會被擋。
- **建議**：在 `<service>` block 內加入 property，value 描述為何此 App 不能歸類於 mediaPlayback / location 等明確類型。

### M3. `READ_EXTERNAL_STORAGE` / `WRITE_EXTERNAL_STORAGE` 在 minSdk 21 下還適用，但 targetSdk 34 已大幅限縮
- **位置**：[AndroidManifest.xml:11-12](Application/LinkBubble/src/main/AndroidManifest.xml:11)
- **建議**：考慮改用 Scoped Storage / `MediaStore` API，或限制 `android:maxSdkVersion="32"`，避免新版裝置實際上拿不到任何權限。

### M4. `org.apache.http.legacy` 仍被 `useLibrary`
- **位置**：[build.gradle:47](Application/LinkBubble/build.gradle:47)、[AndroidManifest.xml:42](Application/LinkBubble/src/main/AndroidManifest.xml:42)

### M6. `MainController.java:415-427` 有偵錯用的 debug TextView 還掛在 WindowManager 上

## 結論

整體升級到 SDK 34 / AndroidX 已經是「能編譯、能跑 debug」的程度，但離「可上架的 release」還有不小距離。最值得優先處理的是 **C1 native lib 缺失**（會在啟動時造成 crash）以及 **C4 缺少 `android:exported`**（Play Store 將拒絕上架）。中期則建議把過時依賴與 `Theme.Holo` 一起換掉，否則之後每次升 targetSdk 都會再痛一次。
